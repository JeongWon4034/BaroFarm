# 외부 API · 생성형 AI 활용 정리 — BaroFarm(바로팜)

> 본 문서는 BaroFarm이 활용한 **외부 API 4종**과 **생성형 AI(LLM/이미지)** 적용 내용을 정리한다.
> 원칙: 가짜 수치를 만들지 않고, 실제 외부 데이터·계산·LLM 응답에 근거한다. 외부 호출 실패 시 항상 **규칙 기반 폴백**으로 서비스가 지속된다.

---

## 1. 외부 API 활용

| API | 용도 | 호출 방식 | 인증 | 비고 |
|---|---|---|---|---|
| **KAMIS 농수산물 시세** | AI 추천가의 **원가 기준선**, 정체상품 소매가, 식료품 트렌드 | RestClient(REST) | 인증키(`KAMIS_CERT_KEY/ID`, 환경변수) | 당일가(dpr1)·1개월전(dpr3)·1년전(dpr4)·단위(unit) 조회. 30분 캐시 |
| **네이버 쇼핑 검색** | AI 추천가의 **경쟁 소매가** 분포·실제 검색 이미지 | RestClient(REST) | Client ID/Secret(환경변수, 로그 비노출) | 핵심명사 매칭 + 가공식품 필터로 동일 품목만. 30분 캐시 |
| **GMS (OpenAI 호환 LLM 게이트웨이)** | 생성형 AI 전반(추천가 정규화·요약·설명·이미지) | OpenAI 호환 chat/image API | API Key(환경변수) | 텍스트 `gpt-4o-mini`(기본)/`gpt-4o`(가격 추론), 이미지 `gpt-image-1` |
| **식약처 조리식품 레시피** | 재고 매칭 레시피 추천·조리법 | RestClient(REST) | 공개 API | 판매중 재고와 재료가 겹치는 레시피 매칭, 재료→상품 연결 |

### 1.1 호출 구조 & 안정성
- **계층**: `KamisClient` · `NaverShoppingClient` · `RecipeApiClient` · `AiClient`(GMS) — Service에서 주입받아 사용.
- **캐싱**: Redis 미사용, **in-memory TTL 캐시**(ConcurrentHashMap, 30분)로 동일 질의 외부 호출 절감.
- **폴백**: 외부망/키 문제로 호출 실패 시 예외를 전파하지 않고 **규칙 기반 결과**로 대체(아래 각 기능 참조). → SSAFY망 밖(클라우드 VM)에서도 핵심 동작 유지.
- **보안**: 모든 키는 `.env`/환경변수로 주입되며 저장소·로그에 노출하지 않는다.

## 2. 생성형 AI(LLM) 활용

게이트웨이: **GMS**(OpenAI 호환). 모델 선택 — 일반 텍스트 `gpt-4o-mini`, **가격 추론은 `gpt-4o`**(단위 파싱 정확도), 이미지 `gpt-image-1`.

| # | 기능 | 모델 | 입력(컨텍스트) | 출력 | 정직성·방어장치 |
|---|---|---|---|---|---|
| ① | **AI 추천 판매가** | gpt-4o | KAMIS 소매시세 + 네이버 경쟁가(묶음단위 포함) | JSON `{price, sellUnit, unitBasis, reason}` — **판매단위 1개당** 추천가 | 묶음단위(2kg/500g/4팩) 파싱·정규화가 핵심. 실패 시 산식 폴백(시세·최저경쟁가 중 낮은 쪽 8%↓) |
| ② | **AI 상품 설명 생성** | gpt-4o-mini | 상품명·카테고리·유통기한 D-day·KAMIS 시세 | 2~3문장 판매 설명 + 사용 컨텍스트 | 시세·임박도 등 실제 값을 프롬프트에 주입 |
| ③ | **구매 분석 인사이트** | gpt-4o-mini | 구매자 주문 집계 KPI | 자연어 요약 + 5글자 소비성향 라벨 + **다음 장보기 추천** | 추천은 **DB 실제 후보 id 중에서만** 선택 → 할루시네이션 차단. 실패 시 할인율 우선 규칙 |
| ④ | **정체 상품 행동추천** | gpt-4o-mini | 며칠째 안 팔린 상품·재고·시세 | 추천가 + 구체 행동(인하·딜전환·번들·노출) | 응답에 엔진 배지(LLM/RULE) 표기로 출처 명시 |
| ⑤ | **판매자 운영 리포트** | gpt-4o-mini | 폐기위험 재고 목록 | 오늘의 핵심 + 추천 행동(예상 폐기손실·AI할인 회수액) | 손실/회수액은 실제 계산값 기반 |
| ⑥ | **식료품 트렌드 카드** | gpt-4o-mini | KAMIS 한달 전 대비 등락 ±3%↑ 품목 | 소비자용 등락 코멘트(오름/내림 각 3개) | 하루 1회 캐시. 실제 시세 등락 기반 |
| ⑦ | **레시피 AI 이미지** | gpt-image-1 | 레시피명·재료 | 완성 요리 이미지 | 하루 캐시. 식약처 조리법과 결합 |

### 2.1 프롬프트 설계 원칙
- **정직성**: "단위 상이 주의", 추천 이유(몇 % 인하·시세 등락)를 함께 노출 — 블랙박스 가격이 아님.
- **구조화 출력**: 가격 추론은 JSON 강제 → `parseIntLoose`·`extractJson`으로 견고 파싱(모델이 `"800원"` 등으로 답해도 복구).
- **할루시네이션 방어**: 구매 추천은 LLM이 자유 생성하지 않고 **DB 실제 상품 후보 집합에서만** 고르도록 제약.

## 3. 데이터 기반 AI (규칙 엔진) — 생성형 AI와 구분

> "AI 폐기예측·동적 할인"은 **LLM이 아니라 결정형 규칙 엔진**(`WastePricingEngine`)이다. 재현 가능·설명 가능하다는 점을 명확히 한다.

- **입력**: 유통기한까지 남은 일수(D-day), 재고
- **산식**: `위험점수 = 0.7·임박도 + 0.3·재고압박` → 위험등급(HIGH/MEDIUM/LOW/EXPIRED), 할인율(최대 60%, 100원 단위 할인가)
- **적용**: 모든 상품 조회 경로 + 주문 시 서버가 권위 있게 재계산
- **데이터 분석**: 판매자 대시보드(Reflex/Streamlit)는 scikit-learn 회귀로 수요/매출 추이를 산출(주문 데이터 기반)

## 4. 요약

| 구분 | 사용 기술 | 핵심 가치 |
|---|---|---|
| 외부 API | KAMIS · 네이버쇼핑 · 식약처 · GMS | 실데이터 기반 가격/레시피/시세 |
| 생성형 AI(LLM) | gpt-4o / gpt-4o-mini / gpt-image-1 (GMS 경유) | 단위 정규화 추천가, 자연어 인사이트, 설명·이미지 생성 |
| 데이터 AI(규칙) | WastePricingEngine, scikit-learn | 설명가능한 폐기 동적 할인·수요 분석 |
| 공통 안정성 | TTL 캐시 + 규칙 폴백 + 키 환경변수화 | 외부 의존 실패에도 서비스 지속·키 비노출 |

**차별점**: 단순히 LLM을 호출한 것이 아니라 — **외부 시세/경쟁가를 LLM이 단위까지 정규화**하고, **추천은 실제 DB 후보로 제약**하며, **폐기 할인은 설명가능한 규칙 엔진**으로 분리해 *정직하고 재현 가능한* AI를 지향했다.
