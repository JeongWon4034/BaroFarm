# 활용 데이터셋 정리 — BaroFarm(바로팜)

> 어떤 데이터를 · 어디서 · 어떻게 썼는지 정리. **원칙: 가짜 수치를 만들지 않는다.**
> 품목·시세 등 사실 데이터는 실제 외부 출처(KAMIS·네이버) 기반이고, 분석용 대규모 유저·주문·행동 데이터는 **통계 규칙 기반 합성**임을 명시한다(합성 데이터에도 할인·절약 계산은 실제 엔진 규칙을 그대로 적용).

---

## 1. 운영 서비스 데이터 (DB 시드)

서비스가 실제로 구동되는 MySQL `freshgrowth`에 적재되는 데이터. 컨테이너 최초 기동 시 `docker-entrypoint-initdb.d`로 자동 적용.

| 파일 | 내용 | 출처 / 생성 방식 |
|---|---|---|
| `backend/.../schema.sql` | 14테이블 DDL + 인라인 시드(회원 3종, 상품 7건, 게시글·댓글, 챌린지 3건) | 직접 작성 (회원 비번 BCrypt) |
| `seed_products_kamis.sql` | 상품 카탈로그(품목·카테고리·가격) | **KAMIS 농수산물 시세 기반** 실제 품목명·시세 |
| `seed_multiseller.sql` | 멀티 판매처(여러 셀러 + 동일 품목 분산) | 가격비교 기능용, KAMIS 품목 기반 |
| `seed_product_images.sql` | 상품 `thumbnail_url`(자체호스팅 이미지) | **네이버 쇼핑 검색**으로 받아 `/seed-images`에 다운로드(외부 핫링크 0) |
| `seed_testdata.sql` | 주문·리뷰·찜·팔로우·챌린지 참여 시나리오 | 기능 시연용 테스트 데이터 |

> 상품·시세 = 실제 KAMIS 기반 / 이미지 = 네이버 검색 자체호스팅 / 시나리오 = 시연용.

## 2. 분석 대시보드용 합성 데이터셋 (`analytics/data/`)

판매자·관리자 분석 대시보드(Reflex·Streamlit)의 **수요예측·매출추이·퍼널 분석**을 보여주기 위한 대규모 데이터. 실제 회원 데이터가 아니라 **통계 규칙으로 합성**했으나, **품목·시세는 실제(KAMIS)**, **할인가는 운영 엔진과 동일 규칙**으로 계산.

| 파일 | 행 수 | 컬럼 | 성격 |
|---|---|---|---|
| `products.csv` | 200 | product_id, seller_id, name, category, price | KAMIS 품목 기반 |
| `users.csv` | 1,000 | user_id, role, name, status, created_at | 합성(페르소나 4그룹) |
| `orders.csv` | 1,320 | order_id, buyer_id, product_id, quantity, total_price, status, is_deal, order_date | 합성(통계 규칙) |
| `reviews.csv` | 610 | order_id, buyer_id, product_id, rating, created_at | 합성 |
| `wishlists.csv` | 2,850 | user_id, product_id, created_at | 합성 |
| `events.ndjson` | 100,215 | event_id, user_id, session_id, ts, event_type, product_id, category, price, is_deal | 합성 행동 로그(퍼널·A/B 분석용) |

### 생성 방식 (`analytics/generate_dummy.py`)
DB의 **실제 품목·폐기기간옵션(lot)을 읽어 참조**하면서 유저·주문만 합성. 적용한 통계 규칙:
1. 페르소나 4그룹 → 카테고리 구매 가중치 차등
2. 일별 주문수 ~ Poisson(λ), 주말 ×1.5
3. 계절성: 품목명 키워드 → 제철 구매확률↑
4. 가격 탄력성: lot 할인가가 쌀수록(폐기 임박) 구매수량↑
5. 딜 선호: 임박(D-3 이내) lot을 더 자주 선택

> **할인가는 백엔드 `WastePricingEngine`과 동일 규칙을 파이썬으로 재현** → 합성 데이터지만 할인·절약액 분석은 실제 서비스 로직과 일치.

### 관련 스크립트
`generate_dummy.py`(합성) · `simulate.py`(시뮬레이션) · `csv_to_seed_sql.py`·`csv_to_orders_seed.py`(CSV→시드 변환) · `seed_*.py`(KAMIS·멀티판매처·engagement 적재) · `fetch_naver_images.py`(이미지 수집)

## 3. 외부 API 원천 데이터 (실시간)

서비스 런타임에 외부에서 받아오는 데이터(저장 X, 30분 TTL 캐시).

| 출처 | 데이터 | 사용처 |
|---|---|---|
| **KAMIS 농수산물 시세** | 당일/1개월전/1년전 소매 시세·단위 | AI 추천가 기준선, 정체상품 소매가, 식료품 트렌드 |
| **네이버 쇼핑 검색** | 경쟁 소매가(lprice)·판매처·상품명·이미지 | AI 추천가 경쟁가 분포, 상품 이미지 |
| **식약처 조리식품 레시피** | 레시피·재료·조리법 | 재고 매칭 레시피 추천 |
| **GMS (LLM)** | gpt-4o/4o-mini 텍스트·gpt-image-1 이미지 | 추천가 정규화·인사이트 요약·설명·이미지 생성 |

## 4. 데이터 정직성 요약

| 구분 | 사실 데이터(실제) | 합성 데이터(통계) |
|---|---|---|
| 품목·카테고리·시세 | ✅ KAMIS | — |
| 상품 이미지 | ✅ 네이버 검색 자체호스팅 | — |
| 할인율·절약액 계산 | ✅ WastePricingEngine 규칙 | — |
| 회원·주문·찜·행동로그(대규모) | — | ✅ 통계 규칙 합성 |

→ **"수치는 실제 계산, 대규모 행동 데이터만 합성"** — 분석 대시보드의 인사이트는 조작이 아니라 실제 규칙·시세에 근거한다.
