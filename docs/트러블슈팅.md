# FreshGrowth 트러블슈팅 기록

프로젝트 진행 중 실제로 막혔던 지점과 해결 방법을 한곳에 모은 문서.
작업기록(`작업기록_2026공통프로젝트.md`)에 흩어져 있던 내용을 정리한 것이며, 실제 겪은 문제만 기록한다.

---

## 1. 로그인 시 jjwt `ServiceConfigurationError` (Java 25)

**증상**
로그인 요청 시 다음 예외 발생, 토큰 발급 실패.
```
java.util.ServiceConfigurationError: io.jsonwebtoken.io.Serializer:
Error accessing configuration file
```

**원인**
백엔드를 **Java 25로 기동**한 것. `JAVA_HOME`이 설정돼 있지 않으면 `/usr/bin/java`가 기본으로 jdk-25를 선택한다.
Spring Boot 3.3.5 + jjwt 0.12.6 조합이 Java 25의 nested-jar ServiceLoader에서 실패한다.

**해결**
반드시 **Java 21**로 기동한다. 재빌드는 불필요(jar 타겟 17).
```bash
export JAVA_HOME=$(/usr/libexec/java_home -v 21)
java -jar target/freshgrowth-backend-0.0.1-SNAPSHOT.jar
```
pom·jjwt 의존성(api/impl/jackson 0.12.6)은 정상이므로 의존성은 건드리지 않는다.

## 2. 앱 기동 불가 — `@MapperScan` 경로 오류

**증상**
앱 시작 시 MyBatis Mapper 빈 등록 실패로 기동 불가.

**원인**
`FreshGrowthApplication.java`의 `@MapperScan("com.freshgrowth.**.mapper")`.
Mapper가 `.mapper` 하위가 아니라 도메인 패키지(user/product/order/review)에 직접 있어 스캔에서 누락됨.

**해결**
```java
@MapperScan("com.freshgrowth")
```
로 변경.

---

## 3. 배치 컴파일 실패 — `RunIdIncrementer` import 경로

**증상**
`batch` 모듈 컴파일 실패.

**원인**
`SeoulMarketPriceJobConfig.java`의 `RunIdIncrementer` import가 Spring Batch 5.2(Boot 3.4+) 경로를 가리킴.
pom은 Boot 3.3.5(Batch 5.1)라 클래스 경로 불일치.

**해결**
```java
import org.springframework.batch.core.launch.support.RunIdIncrementer;
```
로 변경.

## 4. KAMIS API — http 요청 시 302 리다이렉트

**증상**
KAMIS 시세 API를 http로 호출하면 데이터 대신 302 응답.

**원인**
KAMIS는 https 필수. http는 리다이렉트 처리됨.

**해결**
https 엔드포인트 사용.
```
https://www.kamis.or.kr/service/price/xml.do
  ?p_cert_key={key}&p_cert_id={id}&p_returntype=json   (dailySalesList, 소매 01)
```
백엔드에서 직접 동기호출 + 1시간 인메모리 캐시(배치 아님). 인증은 `p_cert_key` + `p_cert_id` 2개.

## 5. 배치 — JSON 대신 HTML 에러페이지 반환

**증상**
배치 실행 시 저장 스텝의 `objectMapper.readTree()`에서 예외 발생, 배치 실패.

**원인**
API 키가 placeholder(미설정) 상태라, 응답이 JSON이 아니라 **HTML 에러페이지**로 내려옴.
유효한 키로 실제 응답을 받아본 적이 없어 파서 필드 매핑(`E_NAME`/`MRKT_DIV`/`PUM_NM_A` 등)도 미검증 상태.

**해결 / 남은 일**
- 유효한 키 입력 후 실제 JSON 구조 1회 확인 → `MarketPriceSaveTasklet.parse()` 필드명 검증·수정.
- 키 없이 테스트하려면 샘플 JSON으로 저장→PDF 스텝 먼저 검증.
- (배치는 현재 '매일밤 폐기예측 + 판매자 AI요약 리포트'로 용도 변경됨)

## 6. 키가 로딩되지 않음 — `application-local.yml` 상대경로 import

**증상**
GMS·KAMIS 키가 주입되지 않아 AI/시세 기능 동작 안 함.

**원인**
키는 gitignore된 `backend/application-local.yml`에서 상대경로(`spring.config.import optional`)로 주입된다.
backend 디렉토리 밖에서 `java -jar`를 실행하면 상대경로 import가 깨져 키가 로딩되지 않음.

**해결**
반드시 **`backend` 디렉토리에서** 실행한다.
```bash
cd backend
java -jar target/freshgrowth-backend-0.0.1-SNAPSHOT.jar
```
키는 절대 채팅·깃에 노출하지 않는다(application-local.yml은 gitignore, example 템플릿만 커밋).

## 7. 인증 전환 — X-USER-ID → JWT

**배경**
초기에는 로그인 후 `userId`를 localStorage에 저장하고 `X-USER-ID` 헤더로 권한을 식별 → 위변조 가능한 임시 방식.

**전환(완료)**
jjwt 0.12.6 도입, `JwtProvider` + `TokenBlacklist`.
로그인 시 토큰 발급, `AuthInterceptor`가 `Authorization: Bearer` 검증, **X-USER-ID 우회 완전 제거**.
로그아웃은 블랙리스트로 토큰 무효화, 역할 불일치 시 403.

**주의**
JWT 전환 후에도 시드 평문 계정은 로그인된다(레거시 `passwordMatches` 호환) — 토큰은 정상 발급되므로 동작에 문제 없음.

## 8. 도커 환경에서 한글 깨짐 (상품명·사용자명 `ê¿€ë°°` 식 모지바케)

**증상**
도커로 띄운 사이트에서 한글이 전부 깨짐. 예: `꿀배 4입` → `ê¿€ë°° 4ìž…`, 메뉴의 `김도연님` → `ê¹€ë„ì—°님`.
(로컬 직접 실행에서는 안 나던 문제 — 도커 MySQL에서만 발생)

**원인 (두 겹)**

1) **DB 적재 시 이중인코딩** — mysql:8.0 컨테이너의 클라이언트 기본 charset이 `latin1`이라,
   `/docker-entrypoint-initdb.d`로 UTF-8 시드(`seed_testdata.sql`)를 넣을 때 한글 바이트를 latin1로
   해석해 이중인코딩된 채 저장됨. 컬럼은 utf8mb4로 정상이지만 **데이터가 깨진** 케이스.

   진단(저장 바이트 확인):
   ```sql
   SELECT name, HEX(name) FROM products LIMIT 1;
   -- 깨짐:   C3AB C2AC C2B4 ...  (이중인코딩)
   -- 정상:   EBACB4 ...           ('무'의 정상 UTF-8)
   SHOW VARIABLES LIKE 'character_set_client';   -- 깨질 때: latin1
   ```

2) **로그인 후 메뉴 이름만 계속 깨짐** — DB를 고친 뒤에도 메뉴의 `OOO님`이 깨져 보이면,
   그건 **DB 고치기 전 로그인 때 `localStorage('user')`에 저장된 깨진 이름**이 남아서임.
   프론트(`stores/auth.js`)는 메뉴 이름을 API 재호출 없이 localStorage 캐시에서 읽음.

**해결**

1) `docker-compose.yml`의 mysql 서비스에 charset 강제 — 서버가 클라 charset(latin1)을 무시하고
   무조건 utf8mb4 사용:
   ```yaml
   mysql:
     image: mysql:8.0
     command:
       - --character-set-server=utf8mb4
       - --collation-server=utf8mb4_unicode_ci
       - --skip-character-set-client-handshake
   ```
2) 이미 깨진 데이터가 볼륨에 남아 있으므로 **볼륨을 비우고 재적재**:
   ```bash
   docker compose down -v && docker compose up --build
   ```
   ⚠️ init 스크립트는 **빈 볼륨일 때만** 실행됨 → compose만 고치고 `up` 하면 안 고쳐짐. `down -v` 필수.
3) 메뉴 이름(2번 원인)은 **재로그인** 또는 브라우저 콘솔 `localStorage.clear()` 후 새로고침으로 갱신.

**검증**: `HEX(name)`이 `EB..`로 시작(정상 UTF-8), `character_set_client = utf8mb4`, API 응답에 `꿀배 4입`.
